The opportunity

OpenSearch excels at search, relevance, and observability.

Either way, customers pay an ETL tax that scales with their data.

The architectural constraint

Single-engine paths hit a wall at fact-table scale.

PPL's subsearch bounds join output to 50 K rows — correct-by-design, so the engine never OOMs. Fact tables have millions of distinct join keys.

30 seconds. Stay calm and technical. The subsearch cap is a safety default (correct-by-design) that stops the engine from OOM'ing on unbounded joins; at fact-table scale (millions of distinct keys) it becomes a ceiling. The remedy is not "remove the cap" — it is federation: route the fact-scale work to the engine built for it.
The fix

Three execution paths. Same PPL query.

Path A All data in OS. Today's default.

Path B Federation. Products in OS, reviews in ClickHouse. PPL pushes the aggregation to CH as a single JDBC query (GROUP BY + ORDER BY + LIMIT).

Path C + IN-list pushdown. The planner drains the bounded left key set and binds it into WHERE parent_asin IN (…) on the CH query at runtime. CH's primary-key index skips >99 % of the fact table.

The customer-facing PPL query is identical in all three. The planner picks the backend and the optimization.

30 seconds. Emphasize: no new syntax, no new datasource call from the customer. They write PPL the same way. The planner decides whether to run aggregation locally or ship it.

The benchmark query (PPL) 

source=products
| where match(title, 'stainless steel cookware')
| sort -_score | head 50
| inner join left=p right=r
    on p.parent_asin = r.parent_asin
  [ source=ch.fed.reviews
    | where timestamp > 1640995200000
    | stats avg(rating) as avg_r,
            count() as n by parent_asin ]

match(title, …) — BM25 search, OS's native strength.
source=ch.fed.reviews — the whole federation delta. That prefix is all the customer writes.

OS's strengths stay strengths.

PPL, BM25, k-NN, Dashboards, observability integration — all unchanged. The search entry point also becomes the federation entry point.

Appendix · for the technically curious

Path C physical plan — IN-list pushdown with CH-side aggregate

Output of POST /_plugins/_ppl/_explain?format=yaml on the benchmark query. 

EnumerableLimit(fetch=[10000])
  EnumerableCalc(expr#0..12=[{inputs}], ... avg_r=[$t0], n=[$t1], parent_asin=[$t2])
    EnumerableHashJoin(condition=[=($2, $4)], joinType=[inner])
      JdbcToEnumerableConverter
        JdbcSort(fetch=[50000])
          JdbcProject(avg_r=[$1], n=[$2], parent_asin=[$0])
            JdbcAggregate(group=[{0}], avg_r=[AVG($3)], n=[COUNT()])
              JdbcFilter(condition=[AND(ARRAY_IN($0, ?0), >($5, 1640995200000))])
                JdbcTableScan(table=[[ClickHouse, ch, fed, reviews]])
      CalciteEnumerableIndexScan(table=[[OpenSearch, products]],
        PushDownContext=[[ FILTER->match(title, 'stainless steel cookware'),
                           SORT->_score desc, LIMIT->50, PROJECT->[features, parent_asin, ...] ]],
        requestedTotalSize=50)
EnumerableHashJoin Rule operand is Join.class — fires on hash, merge, and nested-loop alike, since all three fully scan both inputs. Join-type guard ensures correctness (inner joins always OK; outer joins gated by which side the JDBC input is on).
JDBC subtree (Agg + Filter + Scan) Full right-side plan ships to CH as one SQL, aggregate included. CH returns pre-aggregated rows, not raw reviews.
ARRAY_IN($0, ?0) IN-list sits below the aggregate, so CH filters input rows before GROUP BY. ?0 is runtime-bound to the 50 keys drained from OS. MergeTree PK index skips >99 % of partitions.
Appendix — not part of the main 5-min flow. Flip here when an engineer asks "how does the rule actually rewrite the plan?". Walk the highlights bottom-up: 1. JdbcTableScan — the rule's anchor 2. JdbcFilter with ARRAY_IN — the rule's output, inserted below the aggregate so IN-list prunes input rows before GROUP BY 3. JdbcAggregate — stays inside the JDBC subtree; CH does the aggregation and returns 14 rows, not raw reviews 4. EnumerableHashJoin — operand is Join.class so rule fires on hash / merge / nested-loop alike; all three fully scan both inputs so draining one to populate the other's IN-list is semantically sound for each. If asked about nested-loop specifically: IN-list pushdown actually helps nested-loop the most — cuts O(m × n) to O(m × k) where k is the bounded key count, because the inner side scan is reduced by the primary-key index skip. If asked about LEFT/RIGHT OUTER: the rule has a join-type guard (isJoinTypeCompatibleWithInListPushdown). Directional outer joins preserve unmatched rows on the outer side; filtering those rows via IN-list would produce wrong results, so the rule bails out in those combinations.
