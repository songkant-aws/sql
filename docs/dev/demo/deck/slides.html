Same entry point.
Right engine for each workload.

15 seconds. Set up the tension: OpenSearch customers love OS as their query entry point. This pitch is about what happens when those same customers need analytics on big fact tables.
The problem

OpenSearch customers stay on OS.
Then they hit the analytics wall.

  • PPL, Dashboards, BM25, alerting, observability — all wired up on OS.
  • Customer entry point matters. Customers do not want to switch tools.
  • But analytics on large fact tables — reviews, clickstream, orders — forces them to push fact data into OS.
  • That one decision quietly breaks three things.
30 seconds. OS is the natural place for the customer query layer. Fact data is the natural place for a columnar engine. Today those two things fight each other.

The fix

Three execution paths. Same PPL query.

Path A All data in OS. Today's default.

Path B Federation. Products in OS, reviews in ClickHouse. PPL pushes the aggregation to CH as a single JDBC query (GROUP BY + ORDER BY + LIMIT).

Path C + IN-list pushdown. The planner drains the bounded left key set and binds it into WHERE parent_asin IN (…) on the CH query at runtime. CH's primary-key index skips >99 % of the fact table.

The customer-facing PPL query is identical in all three. The planner picks the backend and the optimization.

30 seconds. Emphasize: no new syntax, no new datasource call from the customer. They write PPL the same way. The planner decides whether to run aggregation locally or ship it.

Where the speedup comes from

Three layers stack cleanly

A
Engine selection. Columnar storage + hash aggregation replaces paginated inverted-index aggregation. OS does 378 pagination round-trips; CH does one shot.
30 s → 600 ms
B
IN-list pushdown. Planner drains 50 bounded keys from the left side and ships them as WHERE parent_asin IN (…). CH's MergeTree primary-key index skips >99 % of partitions.
600 ms → 170 ms
C
Correctness, everywhere below A. Path A's 50 K-row subsearch cap silently drops 96 % of grouped results. Paths B and C don't.
✓

Customer-facing PPL: unchanged. Rule framework is JDBC-dialect-agnostic (Iceberg, Snowflake, others plug in next).

45 seconds. Emphasize "stack cleanly" — each optimization is independent, doesn't fight the others. Also emphasize dialect-agnostic: this demo ships ClickHouse, but the same rules apply to any JDBC fact store.
How it works

One planner, two storages, one PPL

Customer (PPL · Dashboards · BM25) │ ▼ ┌── OpenSearch ──┐ │ │ │ PPL Parser │ │ → Calcite │ │ → Executor │ │ │ │ ┌──────────┐ │ │ │products │ │ │ │ (OS) │ │ │ └──────────┘ │ │ ┌──────────┐ │ │ │JdbcRel │ │ │ │ (CH) │──┼──→ JDBC │ │+IN-list │ │ VPC-private │ └──────────┘ │ └─────────────────┘ │ ▼ ┌── ClickHouse ──┐ │ │ │ MergeTree │ │ pk: (parent_ │ │ asin, ts) │ │ │ │ scans 360 K │ │ rows (not 67M) │ └─────────────────┘

The benchmark query (PPL) 

source=products
| where match(title, 'stainless steel cookware')
| sort -_score | head 50
| inner join left=p right=r
    on p.parent_asin = r.parent_asin
  [ source=ch.fed.reviews
    | where timestamp > 1640995200000
    | fields parent_asin, rating ]
| stats avg(rating) as avg_r,
        count() as n by parent_asin

Calcite rewrites the right-side subtree into a single JDBC query; SideInputInListRule inserts a JdbcFilter with ARRAY_IN($0, ?0); the runtime binder drains the left enumerable's 50 distinct parent_asins and binds them as the ?0 array parameter.

30 seconds. Don't read the PPL out loud. Point at the inner-join subsearch syntax and say "reviews are declared as `ch.fed.reviews` — that prefix is all the customer writes to say 'this table lives in ClickHouse'."
Business implications & the ask

What this means, and what I need

Do customers leave OS? No. OS stays the query entry point, auth surface, dashboards, observability integration. Fact tables just live where they belong.
Lock-in to ClickHouse? No. PplFederationDialect is an extension point — Iceberg, Snowflake, other JDBC targets declare capabilities and plug in.
Readiness? Code complete on feat/ppl-federation. Benchmarked end-to-end on Amazon Reviews 2023 (67 M rows). Reproduction scripts in repo.

  • Greenlight to propose at the next design review.
  • One customer design partner — PPL user with >50 M fact rows they'd rather not keep in OS. Retail and observability both fit.
